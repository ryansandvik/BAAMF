# BAAMF App — Structural Plan (Updated)

## Overview

iOS app for a private 12-member book club. Manages the full lifecycle of each month: host setup, book submissions, vetoes, two-round voting, a reading period, scoring, and historical records. Distributed via TestFlight to fixed members.

---

## Tech Stack

| Layer | Service | Purpose |
|---|---|---|
| Auth | Firebase Authentication | Email/password login; admin role assigned manually |
| Database | Cloud Firestore | Real-time data sync for all app state |
| Book Metadata | Google Books API | Title, author, description, page count, cover image on submission |
| Distribution | TestFlight | Deploy to 12 fixed members |

**Architecture note:** All business logic (vote aggregation, veto threshold checks, round advancement, score finalization) runs client-side. Cloud Functions were planned but not implemented — the host/admin UI gates protect against unauthorized transitions. FCM push notifications are deferred to Phase 7.

## Firestore Data Schema

### `users/{userId}`
```
name:          String
email:         String
role:          "member" | "admin"          // host role is derived from schedule
vetoCharges:   [{ usedAt: Timestamp }]     // max 2 entries; 12-month cooldown each
```

### `months/{monthId}`  *(e.g. `"2026-03"`)*
```
year:                  Int
month:                 Int
hostId:                String
submissionMode:        "open" | "theme" | "pick4"
theme:                 String?
eventDate:             Timestamp?
eventLocation:         String?
eventNotes:            String?
status:                "setup" | "submissions" | "vetoes" | "voting_r1" | "voting_r2" | "reading" | "scoring" | "complete"
selectedBookId:        String?             // document ID of winning book; written when advancing to reading
selectedBookTitle:     String?             // denormalized for card display without extra reads
selectedBookAuthor:    String?
selectedBookCoverUrl:  String?
winningBookId:         String?             // same as selectedBookId; redundant alias for clarity
groupAvgScore:         Double?             // written when host/admin finalizes scoring
```

### `months/{monthId}/books/{bookId}`
```
title:             String
author:            String
description:       String               // from Google Books, or overridden by submitter
pitchOverride:     String?              // submitter's custom pitch (replaces description in UI)
submitterId:       String
googleBooksId:     String
googleRating:      Double?
pageCount:         Int?
coverUrl:          String?
isRemovedByVeto:   Bool                 // Type 1 veto (already read) — immediate removal
vetoType2Voters:   [userId]             // who cast Type 2 vetos on this book
vetoType2Penalty:  Bool                 // true if ≥25% threshold met → -2 net votes in R1
votingR1Voters:    [userId]             // who voted for this book in R1 (arrayUnion/arrayRemove)
votingR2Voters:    [userId]             // who voted for this book in R2
advancedToR2:      Bool                 // set when host advances R1 → R2
```

### `months/{monthId}/scores/{userId}`
```
bookId:       String              // the winning book for that month
scorerId:     String              // same as document ID (userId); kept for convenience
score:        Double              // 1–7, half-points allowed (e.g. 3.5), whole numbers encouraged
updatedAt:    Timestamp
```

## Anonymous Voting Architecture

Votes are stored as arrays directly on book documents (`votingR1Voters`, `votingR2Voters`). Each array contains voter user IDs. Live tallies (array lengths) are visible to all members as the round progresses.

**Privacy note:** Voter identity is visible to anyone who reads the raw Firestore document. This is a pragmatic trade-off for a private 12-member club where trust is assumed. For stronger anonymity, votes would need separate per-user subcollections with restrictive security rules — deferred to future phases if needed.

## Month Status Flow

```
setup → submissions → vetoes → voting_r1 → voting_r2 → reading → scoring → complete
```

- `setup` → `submissions`: Host saves submission mode (via HostSetupView)
- `submissions` → `vetoes`: Host/admin manually advances
- `vetoes` → `voting_r1`: Host/admin opens voting (at the in-person meeting)
- `voting_r1` → `voting_r2`: Host/admin advances; top books flagged `advancedToR2`
- `voting_r2` → `reading`: Host/admin advances; winner's title/author/cover written to month doc
- `reading` → `scoring`: Host/admin advances when the meeting begins
- `scoring` → `complete`: Host/admin finalizes; `groupAvgScore` computed and written to month

All backward transitions clear the relevant data (votes, selections, etc.).

## Development Phases

### Phase 1 — Foundation ✅
Firebase project setup, Auth login flow, user session management, tab navigation shell, Firestore models + service layer.

### Phase 2 — Host Setup & Submissions ✅
Host setup view (submission mode, theme, event info), Google Books API integration, book submission + pitch override, submission list, book edit/swap flows.

### Phase 3 — Vetoes ✅
Type 1 veto (immediate removal), Type 2 veto UI (flag + 25% threshold check), Hard Pass charge tracking with 12-month cooldown, veto reversal and refunds.

### Phase 4 — Voting ✅
Round 1 voting UI (2 votes, anonymous display, vote toggling), `advanceToR2` batch write, Round 2 voting UI (1 vote, vote toggling), reading phase card (shows selected book), portrait orientation lock.

### Phase 5 — Scoring & History  ← CURRENT
Scoring view (1–7, half-points, real-time member scores), finalize scoring (computes group average, advances to complete), history list (all complete months), history detail (full score breakdown per member).

### Phase 6 — Schedule & Admin
Host schedule view, swap request flow, admin "enter scores on behalf of member" panel, admin force-swap.

### Phase 7 — Notifications & Polish
FCM push notification integration, status-change triggers, UI polish, empty states, error handling.

### Phase 8 — Testing & TestFlight
Unit tests for veto threshold + vote counting, UI tests for critical flows, TestFlight build, distribute to all 12 members.

## Key Design Decisions

- **Vote storage:** Arrays (`votingR1Voters`, `votingR2Voters`) on book documents rather than separate per-user subcollections. Simpler reads; privacy trade-off acceptable for a trusted 12-member club.
- **Phase transitions:** All done client-side by host or admin through the gear sheet. No Cloud Functions needed for the current scope.
- **Reading phase:** Interim phase between voting and scoring. Shows the selected book cover/title/author to the club while they read before the meeting.
- **Scoring scale:** 1–7 with 0.5 increments. Whole numbers are encouraged; half-points are allowed but discouraged. Scores are not anonymous — all members see each other's scores.
- **Half-point scores:** `scoreDisplay` extension formats "4" for 4.0 and "4.5" for 4.5 — precision is always visible.
- **Admin "enter on behalf":** Deferred to Phase 6. In Phase 5, only self-scoring is available.
- **Pick-4 mode:** Host submits exactly 4 books; non-hosts see the submission list but cannot submit. The submit button is disabled with an explanatory message.
- **Veto cooldown tracking:** Stored as an array of timestamps on the user doc. Checked before allowing a Type 2 veto.
